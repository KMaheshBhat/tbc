# Implementation Activity Log (2026-01-01)

## Context/Background

Preparing Implementation plan for TBC Health Check (SRE) development activity. After reading the [developer guide](doc/developer-guide.md), [tester guide](doc/tester-guide.md), and [user guide](doc/user-guide.md), I am aligned with the TBC framework principles: technology-agnostic records, git-first approach, plugin-based architecture, and agent-first design. The task is to develop a comprehensive health check script for TBC directory integrity, schema compliance, and link validation to support Site Reliability Engineering for autonomous TBC operations.

## Implementation Plan

Based on the Definition of Done (DoD) outlined in the ejected kernel, the implementation will follow a phased approach to build the TBC Health Check tool. The plan leverages the existing TBC architecture, particularly the `tbc-view` package for indexing and the HAMIFlow system for operations.

### Phase 1: Database & Schema Design (tbc-view Package)
- **Objective**: Establish the persistence layer using SQLite for efficient graph-based health tracking.
- **Tasks**:
  - Define SQLite schema with `nodes` table (id, mtime, hash, type, w1-w5_status) and `edges` table (source, target, type).
  - Integrate `bun:sqlite` into a new `ViewStore` utility class in `packages/tbc-view/src/`.
  - Create database initialization and migration logic.
  - Add TypeScript types for the schema entities.
- **Deliverables**: ViewStore class, schema definitions, database setup utilities.

### Phase 2: Graph-Miner HAMIFlow Implementation
- **Objective**: Build the core mining logic to synchronize filesystem state with the database.
- **Tasks**:
  - Create `GraphMinerFlow` in `packages/tbc-view/src/ops/`.
  - Implement filesystem walking for `mem/`, `sys/`, and `skills/` directories.
  - Add content hashing using Git-style blob hash for change detection.
  - Develop link extraction using regex for internal links `[text](/mem/uuid.md)`.
  - Implement upsert operations to synchronize FS state to SQLite tables.
  - Add watermark calculation logic (W1-W5) during mining.
- **Deliverables**: GraphMinerFlow implementation, link parsing utilities, watermark calculation functions.

### Phase 3: SRE Integrity Auditor Development
- **Objective**: Create the reporting capability for health assessment.
- **Tasks**:
  - Develop `IntegrityReportFlow` in `packages/tbc-view/src/ops/`.
  - Implement queries for broken links (zombie edges), schema violations, and orphan records.
  - Add statistical calculations for system health percentage.
  - Create human-readable report generation with categorized issues.
  - Include repair suggestions for common problems.
- **Deliverables**: IntegrityReportFlow, report generation utilities, health metrics calculations.

### Phase 4: CLI Integration and Testing
- **Objective**: Integrate the health check into the TBC CLI and validate functionality.
- **Tasks**:
  - Register new flows in `apps/tbc-cli/src/ops/`.
  - Add CLI command (e.g., `tbc view audit` or `tbc sys health`).
  - Implement command-line options for different report formats.
  - Create comprehensive tests following TBC testing guidelines (non-destructive, sandboxed).
  - Update relevant SKILL.md files with health check usage instructions.
- **Deliverables**: CLI command implementation, test suite, updated documentation.

### Risk Mitigation
- **Non-destructive Design**: All operations are read-only except for database writes in `dex/`.
- **Incremental Development**: Each phase can be developed and tested independently.
- **Git-first**: All changes committed with meaningful messages.
- **Schema Compliance**: Follow TBC record specifications for any new records created.

## Implementation Notes

- **Architecture Alignment**: The design follows TBC principles by treating the health check as a view operation, using the existing HAMIFlow pattern, and maintaining git-versioned records.
- **Technology Choice**: Using Bun's native SQLite for single-binary deployment, avoiding external dependencies.
- **Watermark Strategy**: The five watermarks provide progressive validation layers, allowing partial health assessment even during development.
- **Extension Points**: The design allows for future extensions like W5 vector status for embedding-based validation.
