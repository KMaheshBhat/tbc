# Sequential Planning Execution Guide (Software Engineering)

## Purpose

This document is a **normative specification** for planning coding / development / software-engineering activities.

It defines **mandatory rules** that MUST be followed by humans and agents when producing execution plans intended to be:
- sequentially executed,
- paused / resumed safely,
- diffed and audited, and
- delegated to agents without semantic loss.

This is not guidance. This is a **hard constraint specification**.

## Normative Keywords

The keywords **MUST**, **MUST NOT**, **SHOULD NOT**, and **MAY** are to be interpreted as described in RFC 2119.

## Core Axiom

> **Execution is linear. Planning MUST be linear.**

Agents and humans may internally reason using graphs or abstractions, but execution proceeds as a **single ordered sequence**.

Any plan that is not linear is invalid.

## Mandatory Design Goals

A compliant plan MUST satisfy ALL of the following:

1. **Replayability**
   - Any step MUST be executable independently.
   - Re-running a step MUST NOT require re-running previous steps.

2. **Interruptibility**
   - Execution MUST be safely stoppable after any step.
   - Inspection and correction MUST be possible before proceeding.

3. **Diff Stability**
   - Semantic changes MUST appear as explicit textual diffs.
   - Reformatting, consolidation, or summarization that hides change MUST NOT occur.

4. **Agent Safety**
   - No step MAY rely on inference, implication, or assumed context.
   - All required information MUST be present inline.

5. **Auditability**
   - A third party MUST be able to determine:
     - intended action
     - execution status
     - remaining work

## Structural Requirements

### 1. Flat Structure (MANDATORY)

- Plans MUST consist of a single ordered list of steps: Step 1, Step 2, …
- Maximum allowed hierarchy depth is **two levels** (Step → bullets).
- Nested sub-steps, trees, DAGs, or branching structures MUST NOT be used.

**Violation Indicator**: Presence of numbered sub-steps (e.g., 4.1, 4.a) or nested headings.

### 2. Incremental Steps Only (MANDATORY)

Each step MUST represent exactly **one unit of progress**.

A step MUST be rejectable if:
- it combines multiple actions or outcomes,
- it represents a phase or category rather than an action,
- it cannot be marked strictly DONE / NOT DONE.

**Violation Indicator**: Conjunctions like "and", "plus", "including" in step titles.

### 3. Executable Binding (MANDATORY)

Each step MUST bind to at least one concrete artifact:
- a command to execute,
- a flow / class / function to implement,
- specific file paths to modify.

Steps without executable bindings MUST NOT exist.

**Violation Indicator**: Steps containing only descriptive language.

### 4. High Inline Information Density (MANDATORY)

All information required to execute a step MUST be included inline.

Plans MUST NOT:
- reference prior conversational context,
- rely on architectural memory,
- defer details to other documents for execution-critical information.

Future executors are treated as **stateless**.

## Required Inline Fields per Step

Each step MUST include, as applicable:
- Objective
- Execution details (commands / flows / files)
- Expected output (inspectable)
- Validation method
- Known failure modes

Omission of required fields is a spec violation.

## Prohibited Patterns (MUST NOT)

### ❌ Step Consolidation

Multiple steps MUST NOT be merged for brevity or readability.

Consolidation causes:
- loss of diff fidelity
- destruction of causal ordering
- unreplayable execution

### ❌ Abstraction Compression

Concrete details MUST NOT be replaced with:
- summaries
- umbrella descriptions
- references such as “as discussed above”.

### ❌ Narrative Planning

Plans MUST NOT be written as prose or documentation chapters.

Explanatory or motivational text belongs outside execution plans.

## Plans as Boundary Objects (MANDATORY)

A compliant plan MUST be safe for use by:
- current-you
- future-you
- other humans
- autonomous agents
- CI / SRE reviewers

Therefore, the plan MUST be:
- explicit
- repetitive where needed
- mechanically checkable

## Approval and Execution Gates

Plans MAY define explicit GO-AHEAD markers.

Execution MUST NOT begin until GO-AHEAD is recorded.

## Failure Modes & Mechanical Detection

This section defines common violations and how they can be detected mechanically (by review or tooling).

### FM-1: Step Consolidation
- **Symptom**: Fewer steps than execution actions
- **Detection**: One step maps to multiple commands or flows
- **Remediation**: Split into atomic steps

### FM-2: Implicit Context Dependency
- **Symptom**: Executor must "remember" earlier discussion
- **Detection**: References to "as above", "previously", or missing execution detail
- **Remediation**: Inline all required information

### FM-3: Non-Executable Steps
- **Symptom**: Step cannot be run or inspected
- **Detection**: No commands, files, or artifacts referenced
- **Remediation**: Bind step to concrete execution artifacts

### FM-4: Diff Smearing
- **Symptom**: Large semantic changes with minimal diff
- **Detection**: Step removal or merging without explicit deletion
- **Remediation**: Restore original steps; apply additive edits only

### FM-5: Hierarchy Leakage
- **Symptom**: Nested plans or branching logic
- **Detection**: Sub-numbering or nested headings
- **Remediation**: Linearize into a single ordered sequence

- Non-negotiable assumptions

## Implementation Plan

### Step 1: {Explicit Action Name}

**Objective**:

**Execution**:
- Command(s) / Flow(s) / File(s)

**Expected Output**:

**Validation**:

**Failure Modes**:

### Step 2: {Next Explicit Action}

(repeat structure verbatim)

### Step N: {Final Explicit Action}

(repeat structure verbatim)

## Development / Execution Workflow

- Build commands
- Development loops
- Environment assumptions

## Testing Strategy

- What is tested
- How it is tested
- What is explicitly NOT tested

## Implementation Notes

(Running execution log. Deviations and decisions.)

## Next / Open Items

- Explicit TODOs
- Ownership

## Deliverables / Outcomes

- Files created
- Commands executed
- State achieved

## Completion Marker

**ALL DONE** — only when every step above is completed and validated.
