## Architectural Design: TBC Write & View (CQRS)

### 1. CLI / UX / AX Layer (Human-System Interface)
The CLI layer captures human intent and maps it to specific internal Command (Write) or Query (Read) orchestrators.

| CLI Command | Architectural Role | Intent | Primary Outcome |
| :--- | :--- | :--- | :--- |
| `tbc mem remember` | **Command** | "Persist this thought/fact." | Minted & templated record in `mem/`. |
| `tbc mem recall` | **Query** | "Find/Project state." | Filtered View Model rendered from `dex/`. |
| `tbc dex rebuild` | **Maintenance** | "Force refresh index." | Full or partial re-scan of collections to update `dex/`. |
| `tbc act start` | **Workflow** | "Open a new context." | Workspace prepared in `act/current/{uuid}`. |
| `tbc act close` | **Workflow** | "Finalize and move on." | Content assimilated via `write-flow`; folder archived. |

### 2. Service & Package Layer (Orchestration)

#### Package: `tbc-write` (The Universal Command Engine)
This package is the authoritative "Command" side of the system. It handles all state mutations regardless of destination.

- **Flow: `write-flow` (fka remember-flow)**
  1. **Resolve Target**: Accepts a collection target (`mem/` or `act/current/{uuid}/`).
  2. **Identity Bridge**: Integrates with `tbc-mint` to ensure every write has a unique, sortable ID.
  3. **Synthesis**: Calls `tbc-system` to hydrate raw content into formal TBC Markdown records.
  4. **Record IO**: Hands off the final payload to `tbc-record-fs` for physical persistence.
  5. **Sync Signal**: Updates the "dirty" watermark in the `dex/` collection to notify the View layer.

#### Package: `tbc-view` (The Universal Query Engine)
This package is the authoritative "Query" side. It manages materialized views and projections for fast retrieval.

- **Flow: `recall-flow`**
  1. **Validation**: Checks if any `tbc-write` watermarks or filesystem timestamps invalidate the current index.
  2. **Index Projection (Dexer)**: Updates the internal index by reading records from both `mem/` and `act/`.
  3. **Search & Filter**: Executes the query against the optimized index.
  4. **View Projection**: Hydrates the search result into a display-ready View Model.

#### Package: `tbc-activity` (Workflow Orchestrator)
A high-level consumer of both `tbc-write` and `tbc-view`. It manages the lifecycle of transient working sessions.

- **Flow: `start-flow`**: Orchestrates folder creation and uses `tbc-write` to initialize the activity context.
- **Flow: `close-flow` (The Assimilator)**: Orchestrates the extraction of knowledge, uses `tbc-write` to finalize it in `mem/`, and handles the archival cleanup.

### 3. Data Flow & Interaction Model

- **Commands** (Write): `tbc-write` -> `tbc-record-fs` -> **Disk Store**
- **Queries** (Read): `tbc-view` -> **Index (Dex)** -> `tbc-record-fs` (Hydration) -> **User**
- **Signals**: `tbc-write` emits a "dirty" signal to the `dex/` collection upon a successful write.

### 4. Shared State Namespaces (Shared Object)
HAMI flows will operate on a partitioned state object to prevent namespace collisions:

- `shared.system`: Read-only rules, root directory paths, and system templates.
- `shared.stage`: Current execution data (Minted IDs, temp messages, transient logic).
- `shared.record`: Data payload destined for or retrieved from the File System.
- `shared.view`: The Materialized View/Projection results for the Query side.

### 5. Design Principles
1. **Symmetry**: `tbc-write` and `tbc-view` represent the clean CQRS split.
2. **Target Agnostic**: `tbc-write` is not tied to a single folder; it is an engine for writing TBC records to any valid collection.
3. **Just-in-Time Indexing**: `tbc-view` ensures consistency by checking for `tbc-write` "dirty" signals before returning query results.
4. **Resilience**: Manual interventions (e.g., editing a file in `mem/` by hand) are supported via `tbc dex rebuild`, which forces the View layer to reconcile with reality.

## Roadmap: TBC 0.4.x - The CQRS Foundation

### 1. The Dex Evolution (tbc-dex)
- **Rename**: Transition `packages/tbc-view` to `packages/tbc-dex`.
- **Digest Protocol**: Implement `.digest.txt` collation.
  - *Target*: `sys/root.md` and `sys/core/*.md`.
  - *Goal*: Single-file system-wide awareness.
- **Referential Protocol**: Implement partitioned `.jsonl` projections.
  - *Partitions*: `dex/{party,goal,...}.jsonl`, `dex/skills.jsonl`, and `dex/act.jsonl` (we may have to think of Activty during last arc).
  - *Format*: One JSON object per line containing UUID, Title, Tags, and FilePath.
- **CLI Integration**:
  - Map `tbc dex rebuild` -> `tbc-dex:rebuild-flow`.

### 2. The Greenfield Writer (tbc-write)
- **Initialization**: Create `packages/tbc-write`.
- **Universal Write Flow**: Orchestrate `tbc-mint` + `tbc-system` + `tbc-record-fs`.
- **Dual-Write Protocol**: 
  - Authoritative Write: Save `.md` to specified collection.
  - Index Write: Append/Update the corresponding `.jsonl` entry in `dex/`.
- **CLI Integration**: Map `tbc mem remember` -> `tbc-write:write-flow`.

### 3. The Universal Viewer (tbc-view)
- **Initialization**: Create fresh `packages/tbc-view`.
- **Recall Flow**: Implement high-speed query logic over the partitioned JSONLs.
- **Just-in-Time Sync**: Add "Dirty Check" logic (mtime comparison) to auto-trigger dex refreshes on recall.
- **Renderers**: Support formatted CLI output (Tables/Lists) from JSONL data.
- **CLI Integration**: Map `tbc mem recall` -> `tbc-view:recall-flow`.

### 4. Activity Overlay (tbc-activity)
- **Refactor**: Transition `tbc-activity` to act as a high-level orchestrator.
- **Persistence**: Use `tbc-write` for all session logs and context updates.
- **The Assimilator**: Implement `act close` to promote activity "souls" to `mem/` via standard `tbc-write` flows.
- **CLI Integration**:
  - Ensure the "Dexer" can traverse nested activity folders to populate `act.jsonl`.
