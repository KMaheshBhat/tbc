## Notes

- Step 1: Design the "namespaces" for the `shared` state
    - the various namespaces
        - `state.stage` → "What am I building right now?"
        - `state.view` → "What does the user need to see?"
        - `state.record` → "What is the truth on the disk?"
        - `state.system` → "What are the rules I must follow?"
      ```
      ┌─────────────────┐
      │     system      │
      ├────────┬────────┤
      │  stage │ view   │
      ├────────┴────────┤
      │     record      │
      └─────────────────┘
      ```
    - we already have got the `shared.record` in use properly.  All `tbc-record*` plugin operations, get/use/put values into it (for most part, there may be few stragglers and dead code, that may need to be reviewed and cleaned up).
    - while the plugin names and shared state names may overlap, a plugin could use more than one states based on the level of abstration.
    - the above diagram is a "rough" shared state namespace adjacency diagram, and not be looked at as dependency diagram.  While record does remain at the core/bottom, the system is more of a combined stage+view state that prepresents the "control" slice of the system.
