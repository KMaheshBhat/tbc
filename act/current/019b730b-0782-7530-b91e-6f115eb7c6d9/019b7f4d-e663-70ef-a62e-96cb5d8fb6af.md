## Review PASS#1 FEEDBACK Addressal Plan

- **FEEDBACK-1 (PROCESS)**: Adopt the "Sequential Planning Execution Guide (Software Engineering)" (located at `act/current/019b730b-0782-7530-b91e-6f115eb7c6d9/sequential_planning_execution_guide_software_engineering.md`) as the mandatory normative specification for all future TBC development activities. This guide enforces linear, incremental planning with high inline information density to prevent context overflow, ensure interruptibility, and maintain auditability. All future plans MUST follow the provided template structure, including explicit steps with objectives, execution details, expected outputs, validation methods, and failure modes. Testing MUST be integrated as dedicated steps after each major implementation phase.
- **FEEDBACK-2 (STRUCTURE:DEVELOPMENT)**: In the next development session, relocate `packages/tbc-view/src/ops/view-store.ts` to `packages/tbc-view/src/store/view-store.ts` to align with architectural separation: `ops/` for HAMIFlow and HAMINode operations, `store/` for CRUD and business logic components. Update all import statements and references accordingly. This maintains consistency with the initial reasoning that ViewStore is not an operation but a data access layer.
- **FEEDBACK-3 (STRUCTURE:DESIGN)**: Maintain watermarks as distinct from node_attributes. Watermarks are dynamic, computed health validation results that change over time based on system state, whereas node_attributes are static properties extracted from record frontmatter. Treating watermarks as attributes would conflate validation metadata with record content, reducing clarity and queryability. Watermarks remain in separate storage for health monitoring purposes.
- **FEEDBACK-4 (STRUCTURE:DESIGN)**: Decompose the monolithic `watermark-executor` HAMINode into five specialized HAMINodes for improved modularity and maintainability: `PresenceWatermarkNode` (file existence and ID validation), `SchemaWatermarkNode` (frontmatter field validation), `StructureWatermarkNode` (content structure checks), `LinksWatermarkNode` (edge integrity validation), and `VectorWatermarkNode` (vector embedding checks). This allows independent testing, parallel execution in future flows, and easier extension. The original node can be deprecated after migration.
- **FEEDBACK-5 (UI/UX)**: Transition CLI commands from `tbc view *` to `tbc dex *` to accurately reflect that these operations expose inDEX (DEX) functionality rather than simple views. Conduct a comprehensive audit of all `dex` and `view` sub-commands across the TBC CLI to ensure consistent naming and functionality alignment. Update command aliases, help text, and documentation to reflect this rationalization. Filesystem-based DEX commands (`tbc dex core|extensions|skills|records`) should be prioritized for consistency.
- **FEEDBACK-6 (FEATURE)**: SQLite natively supports extensions for vector operations, with `sqlite-vec` being the most compatible option for Bun/better-sqlite3 environments (as evidenced by dedicated Bun examples at https://github.com/asg017/sqlite-vec/blob/main/examples/simple-bun/demo.ts and documentation at https://alexgarcia.xyz/sqlite-vec/js.html). The extension can be loaded using better-sqlite3's `loadExtension()` method during ViewStore initialization. If integration proves challenging with Bun's native SQLite driver, consider pivoting to a JavaScript-based vector library (e.g., hnswlib or faiss-js) for in-memory vector operations with SQLite storing only vector metadata. Implement the `VectorWatermarkNode` to validate vector data integrity, and extend the TKG schema to support vector storage and querying. Conduct a proof-of-concept integration test before full implementation to confirm compatibility.
