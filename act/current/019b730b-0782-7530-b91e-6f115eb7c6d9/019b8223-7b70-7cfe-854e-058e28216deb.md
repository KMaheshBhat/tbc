## Review PASS#2 FEEDBACK Addressal Plan

### TBC Architecture Overview
From comprehensive analysis of TBC documentation and codebase:

- **Plugin-Based Architecture**: TBC uses HAMI framework with modular plugins providing atomic operations (nodes) composed into flows
- **Record System**: Plain-text Markdown+YAML records stored in collections (mem/, sys/, dex/, act/)
- **Storage Abstraction**: Currently, `tbc-record-fs` provides filesystem-based record operations (fetch-records, fetch-all-ids, store-records)
- **Consumption Pattern**: Higher-level plugins (tbc-view, tbc-memory, tbc-system) consume record operations via flow composition using `n('tbc-record-fs:fetch-records')` etc.

### Current tbc-view Implementation
- **Direct SQLite Coupling**: ViewStore class directly manages SQLite database operations
- **Watermark System**: Implements 5 watermarks (presence, schema, structure, links, vector) for integrity checking
- **Health Queries**: Provides zombie link detection, orphan records, schema violations
- **Tight Coupling**: tbc-view directly instantiates and manages ViewStore, bypassing TBC's plugin abstraction layer

### CLI Command Structure Analysis
Current commands grouped by domain:
- `sys`: init, upgrade, validate (system lifecycle)
- `dex`: core, records, extensions, skills, index, health, status, audit (indexing + health)
- `gen`: uuid, tsid (ID generation)
- `mem`: companion, prime, stub (memory operations - both queries and record creation)
- `act`: start, backlog, close, show (activity management)
- `int`: probe, generic, gemini-cli, kilocode, goose, github-copilot (interface generation)

### FEEDBACK-7: TBC Record Storage Plugin Architecture

#### Unified Schema Design
Use a single `tbc-record-sqlite` plugin with consistent four-table schema across all databases:

```sql
-- Core entity storage (used in both records.db and meta.db)
CREATE TABLE nodes (
    id TEXT PRIMARY KEY,
    collection TEXT NOT NULL,
    record_type TEXT NOT NULL,
    hash TEXT NOT NULL,
    last_seen_at INTEGER NOT NULL,
    created_at INTEGER,
    file_path TEXT
);

-- Extensible key-value metadata
CREATE TABLE node_attributes (
    node_id TEXT NOT NULL,
    key TEXT NOT NULL,
    value TEXT,
    value_type TEXT NOT NULL,
    updated_at INTEGER NOT NULL,
    PRIMARY KEY (node_id, key),
    FOREIGN KEY (node_id) REFERENCES nodes(id) ON DELETE CASCADE
);

-- Explicit directed relationships
CREATE TABLE edges (
    source_id TEXT NOT NULL,
    target_id TEXT NOT NULL,
    edge_type TEXT NOT NULL,
    created_at INTEGER NOT NULL,
    PRIMARY KEY (source_id, target_id, edge_type),
    FOREIGN KEY (source_id) REFERENCES nodes(id) ON DELETE CASCADE
);

-- Optional metadata for relationships
CREATE TABLE edge_attributes (
    source_id TEXT NOT NULL,
    target_id TEXT NOT NULL,
    edge_type TEXT NOT NULL,
    key TEXT NOT NULL,
    value TEXT,
    PRIMARY KEY (source_id, target_id, edge_type, key),
    FOREIGN KEY (source_id, target_id, edge_type) REFERENCES edges(source_id, target_id, edge_type) ON DELETE CASCADE
);
```

#### Database Separation Strategy
- **`records.db`**: General TBC record indexing (mem/, sys/, dex/ collections)
- **`meta.db`**: System metadata including watermarks, health status, and system state records

#### Watermark Storage as Records
Watermarks stored as records in `meta.db` using the same schema:
- **Node**: Represents the record being watermarked (id = target record ID)
- **Node Attributes**: Store watermark status, message, timestamp, checker
- **Edges**: Link watermarks to their target records or create watermark relationship graphs

Example watermark record structure:
```typescript
{
  id: "019b730b-0782-7530-b91e-6f115eb7c6d9",
  collection: "meta",
  record_type: "watermark",
  attributes: {
    watermark_type: "schema",
    status: 1, // 0=fail, 1=pass, 2=pending, 3=error
    message: "Schema validation passed",
    checked_at: 1704270000,
    checked_by: "tbc-view:graph-miner"
  }
}
```

| Operation | tbc-record-fs | tbc-record-sqlite | Purpose |
|-----------|---------------|-------------------|---------|
| `fetch-records` | ✅ | ✅ | Retrieve records by IDs |
| `fetch-all-ids` | ✅ | ✅ | Get all record IDs from collection |
| `store-records` | ✅ | ✅ | Persist records with metadata |
| `fetch-relations` | ❌ | ✅ | Query relationships between records |
| `store-relations` | ❌ | ✅ | Store relationship data |

#### Database Selection Mechanism
Operations accept database parameter:
```typescript
interface SQLiteStorageOpts {
  database: 'records' | 'meta'; // or custom path
}

// Usage in flows
.next(n('tbc-record-sqlite:fetch-records', { database: 'meta' }))
```

### FEEDBACK-8: CLI Command Restructuring

#### Command Grouping Philosophy
- **User Intent Focus**: Group commands by what users want to accomplish, not implementation details
- **System vs. Data**: Separate system health from data indexing operations
- **Storage Agnostic**: Commands work regardless of underlying storage backend

#### Proposed Command Structure

**`sys` Group (System Lifecycle & Health)**:
- `sys init` - Initialize TBC instance
- `sys upgrade` - Upgrade TBC instance  
- `sys validate` - Validate TBC structure
- `sys health` - Generate SRE integrity report *(moved from dex)*
- `sys status` - Quick health summary *(moved from dex)*
- `sys audit` - Comprehensive system audit *(moved from dex)*

**`dex` Group (Indexing & Read Optimization)**:
- `dex core` - Refresh core system definitions index
- `dex records` - Refresh all records indexes  
- `dex extensions` - Refresh extensions index
- `dex skills` - Refresh skills index
- `dex index` - Index records into configured storage backend *(renamed from index, storage-aware)*

**`mem` Group (Memory Operations - Enhanced)**:
- `mem companion` - Display/query companion information
- `mem prime` - Display/query prime user information  
- `mem stub` - Create stub records *(record creation capability)*
- `mem query` - Advanced memory queries *(future extension)*

#### Storage Backend Integration
- Commands automatically detect available storage plugins
- `dex index` operates on primary storage backend (fs → sqlite migration)
- Health commands query integrity from all configured backends
- Plugin capabilities determine available command features

#### mem Operations Consideration
The `mem` group already demonstrates the hybrid query+creation pattern:
- Query operations: `companion`, `prime` (read from record storage)
- Creation operations: `stub` (write to record storage)

This validates the approach of keeping related operations together rather than splitting by read/write.
