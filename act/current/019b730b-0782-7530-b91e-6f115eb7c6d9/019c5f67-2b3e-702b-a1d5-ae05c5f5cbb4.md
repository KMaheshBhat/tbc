- Step 4: Designing the `tbc-record-sqlite` approach.
  - Discussing with Google Gemini, and preparing [the schema](#sqlite-schema).
  - For now, `from_record_id` `relation_kind='mentions'` `to_record_id` would be the only relation.
  - Discussing with Google Gemini, and preparing [the RDBMSStore interface](#the-rdbmsstore-interface).
  - Further notes and considerations:
    - **CQRS Pattern**: The system follows a Command Query Responsibility Segregation (CQRS) philosophy. The `tbc-record-fs` (DocumentStore) remains the canonical, portable Source of Truth (Write Model), while `tbc-record-sqlite` (RDBMSStore) serves as a high-performance Read Model for graph traversal and complex querying.
    - **Option A Mirroring**: To ensure maximum query performance and system resilience, the RDBMSStore maintains a full content mirror (including the full JSON/Markdown payload). This allows CLI "Recall" and "View" operations to hit the database directly without expensive filesystem tree-walking.
    - **Content Hashing**: The `record` table includes a `content_hash` to optimize the "Mirroring Lifecycle." Parsing for relations/edges only occurs when the hash of the incoming FS record differs from the stored version.
    - **Dependency Inversion**: By defining the `RDBMSStore` interface in the core `@tbc-frameworx/tbc-record` package, Implementation Plugins (like `sqlite`) remain decoupled from TBC-specific domain logic. The Store remains "dumb" (Node/Edge centric), while the Plugins handle the domain-specific mapping (parsing Mentions/Links).
    - **Portable Enrichment (Housekeeping)**: Relations "mined" by the RDBMS can be periodically synced back to the canonical FS front-matter. This ensures that the "Portable Vault" philosophy is maintained—enriched metadata stays with the text even if the database is moved or deleted.

## SQLite Schema

Note: This was initial design in Step 4.  It has been updated since.

```sql
CREATE TABLE IF NOT EXISTS record (
    record_id TEXT PRIMARY KEY,
    record_kind TEXT NOT NULL,
    collection TEXT NOT NULL,
    content_hash TEXT, -- To detect if we need to re-parse relations
    data TEXT NOT NULL DEFAULT '{}', -- Full JSON payload
    created_at TEXT DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at TEXT DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
) STRICT;

CREATE TABLE IF NOT EXISTS record_relation (
    relation_id TEXT PRIMARY KEY,
    relation_kind TEXT NOT NULL, -- e.g., 'mention', 'depends_on'
    from_record_id TEXT NOT NULL REFERENCES record(record_id) ON DELETE CASCADE,
    to_record_id TEXT NOT NULL REFERENCES record(record_id) ON DELETE CASCADE,
    data TEXT NOT NULL DEFAULT '{}',
    valid_from TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    valid_to TEXT, 
    created_at TEXT DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now')),
    updated_at TEXT DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))
) STRICT;

CREATE INDEX idx_record_lookup ON record(collection, record_kind);
CREATE INDEX idx_relation_out ON record_relation(from_record_id, relation_kind);
CREATE INDEX idx_relation_in ON record_relation(to_record_id, relation_kind);
CREATE INDEX idx_relation_temporal ON record_relation(valid_from, valid_to);
```

```typescript
// @tbc-frameworx/tbc-record/src/store-interfaces.ts
export interface RDBMSStore {
  initialize(): Promise<void>;
  teardown(): Promise<void>;
  upsertNode(id: string, kind: string, collection: string, data: Record<string, any>): Promise<void>;
  getNode(id: string): Promise<Record<string, any> | null>;
  deleteNode(id: string): Promise<void>;
  upsertEdge(id: string, kind: string, fromId: string, toId: string, data: Record<string, any>): Promise<void>;
  deleteEdges(fromId: string, kind?: string): Promise<void>;
  getRelatedIds(id: string, direction: 'in' | 'out' | 'both', kind?: string): Promise<string[]>;
  listNodeIds(kind?: string, collection?: string): Promise<string[]>;
}
```
