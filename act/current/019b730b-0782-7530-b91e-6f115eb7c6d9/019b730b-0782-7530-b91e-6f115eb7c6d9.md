## Activity Plan/Approach

### Definition of Done (DoD)

1.  **Persistence Layer**: Implementation of a `tbc-view` SQLite database (`dex/tbc-view.db`) with a Node/Edge schema.
2.  **Watermark Logic**: A Graph-Miner capable of tracking five watermarks:
    - **W1 (Presence)**: FS existance and ID/filename match.
    - **W2 (Schema)**: YAML Frontmatter validity (TBC 0.4).
    - **W3 (Structure)**: Mandatory H2 section presence for temporal logs.
    - **W4 (Relational)**: Extraction of links into the Edge table; detection of "Zombies" (broken links) and "Orphans" (disconnected records).
    - **W5 (Vector Status)**: Placeholder column for embedding status.
3.  **Audit Capability**: A new `tbc-view` HAMIFlow that generates an SRE Integrity Report.
4.  **Single Binary**: Entire execution path remains inside the `tbc-cli` using Bun's native SQLite.

### Phase 1: Database & Schema (tbc-view)
- Define SQLite schema for `nodes` (id, mtime, hash, type, w1-w5_status) and `edges` (source, target, type).
- Integrate `bun:sqlite` into a new `ViewStore` utility within the `tbc-view` package.

### Phase 2: The Graph-Miner (HAMIFlow)
- Create `GraphMinerFlow`:
  - **Walk**: Iterate through `mem/`, `sys/`, and `skills/`.
  - **Hash**: Calculate content hashes (Git-style blob hash) to detect changes regardless of `mtime`.
  - **Parse**: Extract internal links `[text](/mem/uuid.md)` using regex.
  - **Upsert**: Synchronize filesystem state to the SQLite tables.

### Phase 3: SRE Integrity Auditor
- Create an `IntegrityReportFlow` to query the database for:
  - Broken links (Edges pointing to non-existent Nodes).
  - Schema violations (W2/W3 failure flags).
  - Statistical summary of "System Health" percentage.

### Phase 4: CLI Integration
- Register the new flows in `tbc-cli`.
- Map a new command (e.g., `tbc sys health` or `tbc view audit`) to trigger the auditor.
