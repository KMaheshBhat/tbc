# Research Note - SQLite Vector Extension Support for TBC Vector Embeddings

## Context

This research note addresses FEEDBACK-6 from the TBC Health Check (SRE) Review PASS#1, investigating SQLite vector extension capabilities for implementing vector embeddings in the TBC system. The current TBC ViewStore uses Bun's native SQLite driver, and we need to determine the best approach for adding vector search capabilities.

## Current State

- TBC uses Bun's native SQLite driver (not better-sqlite3)
- VectorWatermarkNode exists as a placeholder returning "Not implemented"
- TKG schema has placeholder columns for vector data
- No vector operations are currently implemented

## Research Findings

### SQLite Vector Extensions Overview

SQLite supports extensions for advanced functionality, including vector operations. The most promising option for JavaScript/Bun environments is `sqlite-vec`.

#### sqlite-vec Extension
- **Compatibility**: Designed for better-sqlite3, but has Bun examples
- **Features**: 
  - Vector storage and similarity search
  - Multiple distance metrics (cosine, L2, inner product)
  - Efficient indexing for high-dimensional vectors
- **Documentation**: https://alexgarcia.xyz/sqlite-vec/js.html
- **Bun Example**: https://github.com/asg017/sqlite-vec/blob/main/examples/simple-bun/demo.ts

#### Integration Approaches

**Option 1: sqlite-vec with better-sqlite3**
- Switch TBC ViewStore from Bun's native SQLite to better-sqlite3
- Load sqlite-vec extension using `db.loadExtension()`
- Pros: Full vector functionality, proven compatibility
- Cons: Breaking change from Bun's native driver

**Option 2: JavaScript-based Vector Library**
- Use libraries like hnswlib, faiss-js, or @xenova/transformers
- Store vector metadata in SQLite, perform searches in JavaScript
- Pros: No SQLite extension dependency, stays with Bun native
- Cons: Potentially slower for large datasets, more complex implementation

**Option 3: Hybrid Approach**
- Use sqlite-vec for storage, JavaScript for complex operations
- Implement basic vector operations with extension, advanced features in JS
- Pros: Best of both worlds
- Cons: Most complex to implement

### Technical Considerations

#### Bun Native SQLite Limitations
- No direct extension loading API
- Would require switching to better-sqlite3 or similar
- Current ViewStore uses Bun's Database class

#### Vector Data Schema
Current TKG schema has placeholder vector columns. Need to extend:
```sql
-- Vector storage table
CREATE TABLE vectors (
  node_id TEXT PRIMARY KEY,
  vector BLOB, -- or use sqlite-vec specific types
  dimensions INTEGER,
  created_at INTEGER,
  updated_at INTEGER
);

-- Vector index (with sqlite-vec)
CREATE VIRTUAL TABLE vector_index USING vec0(
  node_id TEXT,
  vector FLOAT[384] -- example dimension
);
```

#### Performance Expectations
- sqlite-vec: Fast for moderate datasets (< 1M vectors)
- JavaScript libraries: Better for very large datasets or complex operations
- Hybrid: Optimal balance for TBC use case

## Recommendations

### Primary Recommendation: Hybrid Approach
1. **Short-term**: Implement basic vector storage using JavaScript library (hnswlib-node)
2. **Medium-term**: Evaluate sqlite-vec integration if performance needs arise
3. **Implementation Plan**:
   - Extend ViewStore with vector methods
   - Implement VectorWatermarkNode for integrity checks
   - Add vector indexing flows
   - Test with sample embeddings

### Proof-of-Concept Requirements
- Create test vectors (e.g., 384-dimension embeddings)
- Implement basic similarity search
- Measure performance vs. accuracy trade-offs
- Validate with real TBC content (if available)

### Risks and Mitigations
- **Extension Compatibility**: Test thoroughly in isolated environment
- **Performance**: Start with small datasets, scale gradually
- **Maintenance**: Choose well-maintained libraries

## Next Steps

1. Implement basic vector storage using JavaScript library
2. Create proof-of-concept with sample data
3. Extend VectorWatermarkNode for actual validation
4. Performance testing and optimization
5. Documentation and integration testing
